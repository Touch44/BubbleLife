## v6.9.5 — 2026-05-31

## v6.9.4 — 2026-05-31

### Bug Fix: Tag chip misleading "click to open" tooltip
- Tag chips always showed tooltip "Click label to open tag · × to remove" even when the tag is a plain string label with no corresponding tag entity in the database
- Clicking the label did nothing (silently failed) if no entity existed
- **Fix:** Default tooltip is now "× to remove" only; cursor is `default` by default; clicking the label checks if a tag entity exists — if yes, opens the panel and updates tooltip/cursor for subsequent clicks; if no entity, shows toast: `"tagname" is a label only — no tag entity exists yet`

## v6.9.3 — 2026-05-31

**Project view bug fixes + health score improvements**

### Bug Fix: Templates panel rendering twice
- `context:changed` event fires during first-time setup, triggering a second concurrent `renderProjects()` while the first is suspended at `await _loadUserTemplates()`
- Both async calls resolved and each appended a sidebar+editor panel to `el`
- **Fix:** Added `_renderSeq` counter (incremented each call); `mySeq !== _renderSeq` guard aborts stale renders before `el.innerHTML = ''` and inside `_renderTemplatesView` before appending

### Improvement: Health score for projects with 0 tasks
- A newly created project with no tasks was showing a numerical score (e.g. 66) — misleading since "healthy with no tasks" is meaningless
- **Fix:** Projects with 0 tasks now show `—` ring (grey, hollow) with tooltip "No tasks yet — add tasks to track health"
- Signal 3 partial credit reduced from +15 to +8 for 0-task projects
- Signal 5 (overdue ratio): 0-task projects now score 0 instead of 25 — can't claim "no overdue tasks" when there are no tasks at all
- Net effect: a new 0-task project scores ~33 (setup bonus only) instead of 66, and the ring is hidden until tasks are defined

## v6.9.0 — 2026-05-31

**Major bug audit — 30 bugs fixed, 8 efficiency improvements**

### Critical Bug Fixes
- **[B01/B08]** Removed dead `deleteEntity` import and broken `Promise.all([async()=>{_getDB}()])` in ENTITY_DELETED handler

- **[B04]** Family Wall now lazy-loads Related panels via IntersectionObserver (200px prefetch margin) — prevents 20+ concurrent `getSuggestions` calls on initial render
- **[B05]** Incremental entity updates route through main thread only — eliminates race where `KLRE_INDEX_UPDATED` emitted before Worker finished writing to IDB
- **[B06]** Worker `handleUpdateEntity` now uses 1-hour corpus cache (`_workerCorpus`) — was O(N) corpus rebuild per entity save (300 IDB reads per keystroke)
- **[B07]** `confirmSuggestion` now invalidates both `fromId` and `toId` suggestion caches
- **[B08]** ENTITY_DELETED handler cleaned — dead `Promise.all` removed
- **[B09]** `accessesA = _accessWindow.filter()` pre-computed once before candidate loop — was recomputed 200× per `getSuggestions` call (10,000 → 50 comparisons)

### Significant Bug Fixes
- **[B10]** Settings toggles pre-set to defaults before async IDB reads — eliminates unchecked→checked flicker
- **[B11]** `invalidatePulseCache` now conditional — only fires for task/event/tagged entity saves, not all saves
- **[B12]** `klre_enabled` and `klre_min_score` cached in memory (`_cachedEnabled`/`_cachedMinScore`) — saves 2 IDB reads per `getSuggestions` call; `invalidateSettingsCache()` called on settings change
- **[B14]** STALE_TASK uses `updatedAt` consistently (was mixing `createdAt`/`updatedAt`)
- **[B15]** Dismiss `row.remove()` guarded with `row.isConnected` — prevents throw on already-removed elements
- **[B16]** Expand toggle now calls `renderRelatedPanel()` for fresh data — was re-rendering stale closure items
- **[B17]** Deleted entities and `taskInstance` types filtered from `allEntities` before 3 parallel batch preloads
- **[B18]** Calendar related panel guards with `relEl.isConnected` — prevents rendering into detached node after popover closes
- **[B19]** Search clustering BFS uses `resultMap.get()` O(1) — was O(N²) `results.find()` inside BFS loop
- **[B20]** Worker message handler adds `.catch()` — unhandled rejections now report `INDEX_ERROR` to main thread

### Minor Fixes
- **[B27]** `getEntityTypeConfig` cached per type in `_typeConfigCache` — eliminates redundant calls during corpus build
- **[B29]** Knowledge Pulse badge `className` trimmed — no trailing space when `pulseType` unknown
- **[B30]** `renderKnowledgePulse` checks `isConnected` before rendering — prevents work on detached container

### Efficiency Improvements
- **[E1]** Family Wall IDB pressure ~20× lower via lazy loading
- **[E2]** Worker corpus rebuild: ~300× faster on warm cache
- **[E3]** 2 IDB settings reads saved per `getSuggestions` call via in-memory cache
- **[E4]** ~190 redundant `getEntityTypeConfig` calls eliminated per index build
- **[E5]** Deleted entities, taskInstance, and focal entity excluded from 3 pre-load batches
- **[E6]** Co-access window scan: 10,000 comparisons → 50 per call
- **[E7]** BFS lookup O(N²) → O(1) Map
- **[E8]** Named entity boost (3×) now applies correctly on incremental updates (was always `new Set()`)
